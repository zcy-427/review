**1.进程同步与互斥**

同步与互斥解决的是：
多个进程并发执行时，如何正确、安全地共享资源，并按需要的顺序协作。

**互斥**：多个进程在访问临界资源时，任意时刻最多只允许一个进程进入临界区。
**同步**：多个进程为了完成共同任务，在执行次序上需要相互等待、相互配合。

**同步vs互斥**

| 对比      | 互斥          | 同步            |
| ------- | ----------- | ------------- |
| 解决问题    | 不能同时访问资源    | 必须按顺序协作       |
| 关键词     | 不能同时        | 先后顺序          |
| 常见资源    | 打印机、缓冲区、变量  | 生产后消费、读后写     |
| 信号量常见初值 | 1           | 看条件，常为 0      |
| 例子      | 两个进程不能同时写文件 | 生产者先生产，消费者后消费 |

**临界资源与临界区**

临界资源：一次只允许一个进程使用的资源，称为临界资源
临界区：每个进程中访问临界资源的那段代码，称为临界区。

**2.PV操作**

**信号量**：信号量可以理解为一个特殊整数，用来表示资源数量或某种条件是否满足。
- 资源型信号量：比如有 3 台打印机：

```text
printer = 3
```

表示可用打印机数量为 3。

- 互斥信号量：如果某资源一次只能一个进程用：

```text
mutex = 1
```

表示当前可以进入临界区。

**P操作**：可以理解为：

```text
申请资源 / 等待条件
```

通俗说：

> 我要进去/我要用资源，如果没有，就等。

常见写法：

```c
P(S)
```

含义：

```text
S = S - 1
如果 S < 0，则进程阻塞
```

P 是申请，可能阻塞
```

**V操作**：可以理解为：

```text
释放资源 / 通知条件满足
```

通俗说：

> 我用完了/事情做完了，通知别人。

常见写法：

```c
V(S)
```

含义：

```text
S = S + 1
如果 S <= 0，则唤醒一个等待进程
```

**互斥问题模板**：
多个进程不能同时进入临界区
```c
semaphore mutex = 1;

进程 Pi:
    P(mutex);
    临界区;
    V(mutex);
```

**同步问题模板**：
要求 A 操作必须先于 B 操作执行。
```c
semaphore S = 0;

进程 A:
    A操作;
    V(S);

进程 B:
    P(S);
    B操作;
```

**3.生产者-消费者问题**
生产者—消费者问题解决的是：

> 多个生产者和消费者并发访问有限缓冲区时，如何保证不乱放、不乱取、不同时修改缓冲区。

把缓冲区想象成一个货架。

- 生产者 = 工厂工人，负责往货架放商品
- 消费者 = 顾客，负责从货架拿商品
- 缓冲区 = 货架
- 货架容量有限，比如最多放 n 个商品

必须满足三件事：

| 条件     | 含义          |
| ------ | ----------- |
| 货架满了   | 生产者不能再放，必须等 |
| 货架空了   | 消费者不能再拿，必须等 |
| 放/拿货架时 | 一次只能一个人操作货架 |
**正式定义**：若干生产者进程和消费者进程共享一个有限缓冲区，生产者向缓冲区放入产品，消费者从缓冲区取出产品。需要通过同步机制保证生产者在缓冲区未满时生产，消费者在缓冲区非空时消费，并通过互斥机制保证对缓冲区的访问互斥进行。

**3个信号量**：
假设缓冲区大小为 `n`。

| 信号量     | 初值  | 含义      |
| ------- | --- | ------- |
| `mutex` | 1   | 互斥访问缓冲区 |
| `empty` | n   | 空缓冲区数量  |
| `full`  | 0   | 已有产品数量  |
**模板**：
```c
semaphore mutex = 1;
semaphore empty = n;
semaphore full = 0;

Producer:
while (true) {
    生产产品;
    P(empty);
    P(mutex);
    放入产品;
    V(mutex);
    V(full);
}

Consumer:
while (true) {
    P(full);
    P(mutex);
    取出产品;
    V(mutex);
    V(empty);
    消费产品;
}
```

**4.读者—写者问题**
读者—写者问题解决的是：

> 多个进程共享同一数据文件时，如何允许多个读者同时读，但保证写者写入时独占访问。
```text
读读共享，读写互斥，写写互斥。
```

**正式定义**：读者—写者问题是指多个读进程和写进程共享同一数据对象，允许多个读进程同时访问该对象，但写进程访问时必须互斥，即写进程与其他读进程或写进程不能同时访问共享对象。

| 变量 / 信号量    | 初值  | 作用                         |
| ----------- | --- | -------------------------- |
| `readcount` | 0   | 当前正在读的读者数量                 |
| `mutex`     | 1   | 保护 `readcount`，防止多个读者同时修改它 |
| `rw`        | 1   | 控制读者和写者对共享文件的互斥访问          |
**标准读者优先 PV 模板**
**Reader 读者进程**

```c
Reader:
while (true) {
    P(mutex);
    readcount++;
    if (readcount == 1)
        P(rw);
    V(mutex);

    读文件;

    P(mutex);
    readcount--;
    if (readcount == 0)
        V(rw);
    V(mutex);
}
```

**Writer 写者进程**

```c
Writer:
while (true) {
    P(rw);
    写文件;
    V(rw);
}
```

**为什么只有第一个读者执行 `P(rw)`？**

```text
因为读者之间可以共享访问文件，不需要相互互斥。只要第一个读者执行 `P(rw)`，就可以阻止写者进入；后续读者可以直接一起读，不必再执行 `P(rw)`。
```
**为什么只有最后一个读者执行 `V(rw)`？**
```text
只要还有读者在读，写者就不能进入。因此只有当 `readcount == 0`，也就是最后一个读者离开时，才能执行 `V(rw)` 释放共享文件，允许写者进入。
```

**5.哲学家进餐问题**

**常见的解决方法**：

|方法|思想|
|---|---|
|最多允许 4 个哲学家同时拿筷子|破坏循环等待|
|奇偶编号哲学家拿筷子顺序不同|破坏循环等待|
|只有左右筷子都可用时才允许拿|避免占有一根等另一根|
**方法1模板:**
```c
semaphore room = 4;
semaphore chopstick[5] = {1, 1, 1, 1, 1};

Philosopher i:
while (true) {
    思考;

    P(room);
    P(chopstick[i]);
    P(chopstick[(i + 1) % 5]);

    吃饭;

    V(chopstick[(i + 1) % 5]);
    V(chopstick[i]);
    V(room);
}
```

**6.死锁**

**定义**：死锁是指多个进程在运行过程中因争夺资源而造成的一种僵局。若无外力作用，这些进程都将无法继续向前推进。

**4个条件**：

| 条件      | 含义                 |
| ------- | ------------------ |
| 互斥条件    | 资源一次只能被一个进程使用      |
| 占有并等待条件 | 进程已经占有资源，同时又等待新的资源 |
| 不可剥夺条件  | 已分配给进程的资源不能被强行夺走   |
| 循环等待条件  | 存在一组进程形成首尾相接的等待链   |

**死锁处理方法**
- 死锁预防：通过破坏死锁四个必要条件之一，使死锁不可能发生
- 死锁避免：在资源分配过程中动态判断，如果分配后可能进入不安全状态，就不分配
- 死锁检测：系统允许死锁可能发生，定期检查系统中是否已经发生死锁。
- 死锁解除：检测到死锁后，采取措施打破死锁。

| 方法   | 发生时机 | 核心思想       | 例子        |
| ---- | ---- | ---------- | --------- |
| 死锁预防 | 事前   | 破坏四个必要条件之一 | 资源有序分配    |
| 死锁避免 | 分配时  | 判断是否安全     | 银行家算法     |
| 死锁检测 | 事后   | 检查是否发生死锁   | 资源分配图检测   |
| 死锁解除 | 死锁后  | 打破死锁状态     | 撤销进程、剥夺资源 |
